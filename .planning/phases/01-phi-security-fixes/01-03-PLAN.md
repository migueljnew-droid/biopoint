   - `export function shouldProcessModel(model: string): boolean` -- checks if model is in ENCRYPTED_FIELDS map
   - `export async function encryptDataObject(model: string, data: any): Promise<any>` -- encrypts fields for a given model's data object

**Important: $extends type change.**
`basePrisma.$extends({...})` returns a DIFFERENT TypeScript type than `PrismaClient`. The export from `db/src/index.ts` will no longer be typed as `PrismaClient`. Anywhere that imports `prisma` and uses it should work fine (duck typing), but any function typed as `(prisma: PrismaClient) => ...` will get a type error. The main instance of this is `setupEncryptionMiddleware` itself (which we are removing) and potentially the test file. Update types accordingly:
- Export the extended client type: `export type ExtendedPrismaClient = typeof prisma;`
- If any imports break due to type mismatch, update the import to use `ExtendedPrismaClient` or `typeof prisma`.

**Step C: Build the extended Prisma client in db/src/index.ts**

1. Open `db/src/index.ts`.
2. Import the encryption functions from the refactored encryption.ts:

   Note: The import path must be correct relative to `db/src/`. If this cross-package import is problematic (monorepo boundary), move the encryption utility functions to `db/src/encryption.ts` or a shared location. Use whatever path structure works with the existing monorepo setup.

3. Replace the bare `new PrismaClient()` with:
   ```typescript
   const basePrisma = new PrismaClient({
     log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
   });

   export const prisma = basePrisma.$extends({
     query: {
       $allModels: {
         async create({ model, args, query }) {
           if (shouldProcessModel(model)) {
             args.data = await encryptDataObject(model, args.data);
           }
           return query(args);
         },
         async createMany({ model, args, query }) {
           if (shouldProcessModel(model) && Array.isArray(args.data)) {
             args.data = await Promise.all(args.data.map(d => encryptDataObject(model, d)));
           }
           return query(args);
         },
         async update({ model, args, query }) {
           if (shouldProcessModel(model)) {
             args.data = await encryptDataObject(model, args.data);
           }
           return query(args);
         },
         async upsert({ model, args, query }) {
           if (shouldProcessModel(model)) {
             if (args.create) args.create = await encryptDataObject(model, args.create);
             if (args.update) args.update = await encryptDataObject(model, args.update);
           }
           return query(args);
         },
         async findMany({ model, args, query }) {
           const results = await query(args);
           if (shouldProcessModel(model) && Array.isArray(results)) {
             return Promise.all(results.map(r => decryptRecord(model, r)));
           }
           return results;
         },
         async findFirst({ model, args, query }) {
           const result = await query(args);
           if (shouldProcessModel(model) && result) return decryptRecord(model, result);
           return result;
         },
         async findUnique({ model, args, query }) {
           const result = await query(args);
           if (shouldProcessModel(model) && result) return decryptRecord(model, result);
           return result;
         },
       },
     },
   });

   export type ExtendedPrismaClient = typeof prisma;
   ```

1. **Wire connection pool into Prisma client (SEC-06):**

   Open `db/src/index.ts` (already modified in Task 1).

   The recommended approach per research is to append `connection_limit` and `pool_timeout` as URL query parameters to `DATABASE_URL`. This works with Neon's connection pooler.

   Add a helper function before the PrismaClient instantiation:
   ```typescript
   function getConnectionUrl(): string {
     const url = process.env.DATABASE_URL || '';
     if (url && !url.includes('connection_limit')) {
       const separator = url.includes('?') ? '&' : '?';
       const limit = process.env.NODE_ENV === 'production' ? 20 : 5;
       return `${url}${separator}connection_limit=${limit}&pool_timeout=15`;
     }
     return url;
   }
   ```

   Then update the `basePrisma` constructor:
   ```typescript
   const basePrisma = new PrismaClient({
     log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
     datasources: {
       db: { url: getConnectionUrl() },
     },
   });
   ```

   This wires the pool config from `apps/api/src/config/database.ts` values into the actual client. The database.ts `getPrismaConfig()` function is effectively dead code now (the URL param approach is simpler and officially recommended by Neon). Leave a comment noting that `getPrismaConfig()` in database.ts is superseded by URL query params here.
