<objective>
Fix two security bugs: (1) replace the `[DECRYPTION_FAILED]` sentinel string with null + server-side error logging, and (2) remove the `if (results.length > 0)` guards that skip audit logging for empty-result PHI queries.

Purpose: The sentinel string leaks encryption internals to API consumers. The conditional audit logging violates HIPAA's requirement to log all access attempts, not just successful data retrievals.
Output: Patched encryption.ts with proper error handling, and 8 audit log call sites across 4 files fixed to log unconditionally.
</objective>

  <action>
1. Open `apps/api/src/middleware/encryption.ts` and find the decryption error handler (around line 188-207, the `decryptSingleItem` or equivalent function).

2. Find EVERY occurrence of `'[DECRYPTION_FAILED]'` in the file. There may be more than one catch block that sets this sentinel.

3. Replace each occurrence with:
   ```typescript
   catch (error) {
     // Log the error server-side for investigation -- never expose encryption details to clients
     console.error(`[ENCRYPTION] Decryption failed for ${model}.${field}:`,
       error instanceof Error ? error.message : 'Unknown error');
     // Return null instead of sentinel string (SEC-03)
     decryptedItem[field] = null;
     // Remove the broken encrypted field from the response
     if (`${field}_encrypted` in decryptedItem) {
       delete decryptedItem[`${field}_encrypted`];
     }
   }
   ```

4. Also search for any OTHER sentinel strings or error markers that might leak encryption internals (e.g., `'ENCRYPTED'`, `'ERROR'`, `'CORRUPTED'`). Replace any found with the same null + log pattern.

5. Verify that the function signature return types are compatible with returning `null` for decrypted fields. If the return type is `string`, change it to `string | null`. This aligns with how optional PHI fields already work in the Prisma schema.

6. Do NOT change the encryption (write) side -- only the decryption (read) error handling.
